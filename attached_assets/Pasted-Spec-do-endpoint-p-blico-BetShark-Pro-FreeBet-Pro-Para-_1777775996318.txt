Spec do endpoint público — BetShark Pro → FreeBet Pro
Para quem é este documento: equipe técnica do Grupo Shark 100% Green / BetShark Pro.

Objetivo: definir o contrato do endpoint público que o painel BetShark Pro vai chamar para criar, atualizar, arquivar e definir resultados de procedimentos diretamente no catálogo público do FreeBet Pro, fechando a paridade §8.5 que ficou em aberto.

Pré-requisito de leitura: 01-contexto-procedimentos-pro.md. Toda a regra de negócio (1-para-N, status_operacao, modelo aditivo de resultado, SSE) descrita lá vale aqui também — este documento só descreve como vocês entram nesse fluxo de fora.

Status: aprovada (2026-05-03). A BetShark devolveu as respostas dos 8 pontos abertos da §13 e propôs ajustes pequenos de mapeamento — todos absorvidos abaixo. Implementação real é uma task separada (Task #560), não está no escopo deste doc.

1. Visão geral
A BetShark Pro vai chamar um conjunto de rotas HTTP REST hospedadas no domínio do FreeBet Pro. Cada chamada autenticada cria/atualiza um procedimento no nosso catálogo público (procedimentos_tarefas, com origem='CATALOGO' e publico=true) como se um gerente nosso tivesse cadastrado pelo Painel de Envios.

A partir do momento em que a chamada é aceita, todo o resto do fluxo descrito no doc 01 acontece automaticamente:

O procedimento aparece em <2s para todos os clientes Pro elegíveis (publicação 1-para-N via SSE).
Cada cliente Pro mantém o seu status pessoal (PENDENTE/VOU_FAZER/FEITO/DISPENSEI).
Quando vocês mandarem o resultado, o status_operacao do card é recalculado (LUCRO_DIRETO ou FALTA_GIRAR_FB) seguindo a mesma regra do nosso modal "Definir Resultados".
BetShark Pro (Supabase)              FreeBet Pro (este endpoint)             Clientes Pro
─────────────────────────            ───────────────────────────             ──────────────
Gerente Shark cadastra      ──HTTP──►  POST /api/integracoes/betshark
no painel deles                        /procedimentos
                                            │
                                            ▼
                                       INSERT em procedimentos_tarefas
                                       (origem=CATALOGO, owner=Shark)
                                            │
                                            ▼
                                       notifyProcedimentoChange (SSE) ──►   Card aparece
                                                                            no Pro <2s

2. Identidade do parceiro
Cada parceiro de integração (no MVP, só a BetShark Pro) é cadastrado uma vez no nosso lado. A configuração mora numa nova tabela integracoes_parceiros:

Campo	Tipo	Significado
id	serial	Primary key.
slug	text unique	Identificador legível. Sugerido: 'betshark'.
nome	text	Nome amigável ('BetShark Pro — Grupo Shark 100% Green').
owner_client_id	int FK clients.id	Qual catálogo Pro recebe os procedimentos. Toda chamada cai aqui — fica idêntico a um gerente do owner ter cadastrado pelo Painel de Envios.
api_key_hash	text	Hash (bcrypt) da API key. Nunca guardamos a chave em claro.
webhook_secret	text	Segredo HMAC, gerado uma vez e guardado em variável de ambiente do lado do parceiro. Usado pra assinar o body.
ip_allowlist	jsonb	Array de IPs/CIDR. Vazio = sem restrição.
rate_limit_rpm	int	Default 60. Limite de chamadas por minuto por API key.
ativo	bool	Kill switch operacional.
created_at, updated_at	timestamptz	Auditoria.
Como a chave chega até vocês: um Super Admin nosso vai gerar a chave uma vez no painel interno, copiar uma única vez o valor em claro, e mandar pra Shark por canal seguro (1Password, Bitwarden ou similar). A partir daí só temos o hash. Se vocês perderem, geramos uma nova (revoga a antiga).

3. Endpoints
Base: https://app.freebetpro.com.br/api/integracoes/betshark

Nota: o domínio final pode mudar (homolog vs prod). A spec define caminhos relativos; o domínio final fica no envelope de integração que mandamos junto com a chave.

Método	Rota	O que faz
POST	/procedimentos	Cria um procedimento, ou faz upsert se o external_id já existe.
PATCH	/procedimentos/:external_id	Atualiza campos parciais. Whitelist server-side.
PATCH	/procedimentos/:external_id/resultado	Define o resultado real (modal "Definir Resultados").
POST	/procedimentos/:external_id/arquivar	Arquiva (soft delete) ou restaura.
GET	/procedimentos/:external_id	Lê o estado atual no nosso lado. Útil pra reconciliação.
:external_id é o UUID do procedures.id do lado da BetShark — é a chave que liga os dois sistemas.

4. Autenticação
Toda requisição precisa de três headers (ausência de qualquer um → 401):

Authorization: Bearer <api_key>
X-Integracao-Slug: betshark
X-Timestamp: 1746290520
X-Signature: sha256=<hex>

4.1 Bearer
api_key é a string em claro que entregamos uma única vez. Comparada via bcrypt.compare contra api_key_hash no banco.
O X-Integracao-Slug é redundância proposital (sanity check, evita confundir parceiros se um dia tiver mais de um).
4.2 Janela de timestamp
X-Timestamp é o unix epoch em segundos (UTC) no momento da requisição.
Aceitamos uma janela de ±300s (5 min). Fora disso, 401. Isso impede replay de chamadas antigas.
4.3 HMAC do body
Calculado assim, do lado de vocês, antes de enviar:

mensagem  = X-Timestamp + "." + corpo_cru_bytes
assinatura = HMAC_SHA256(webhook_secret, mensagem)
X-Signature = "sha256=" + hex(assinatura)

Do nosso lado: rejeitamos com 401 se a assinatura recalculada não bater. Esse é o mesmo padrão dos webhooks que já recebemos da Lastlink e do FrontZapp — então a abordagem está validada em produção.

4.4 Erros de auth
Sempre 401 com body genérico:
{ "ok": false, "error": { "code": "UNAUTHORIZED", "message": "auth failed" } }

Não detalhamos o motivo (chave errada vs assinatura errada vs janela expirada) pra não dar pista pra brute-force.
5. Idempotência
A chave de idempotência é o external_id (UUID do procedures.id da BetShark). Vamos persistir esse valor numa coluna nova:

ALTER TABLE procedimentos_tarefas
  ADD COLUMN external_id TEXT,
  ADD COLUMN external_source TEXT;
CREATE UNIQUE INDEX procedimentos_tarefas_external_uniq
  ON procedimentos_tarefas (external_source, external_id)
  WHERE external_id IS NOT NULL;

POST /procedimentos com external_id que não existe → cria nova linha. Resposta 201 Created com created: true.
POST /procedimentos com external_id que já existe (mesmo external_source='betshark') → trata como update (upsert). Resposta 200 OK com created: false.
Resultado prático: vocês podem reenviar o mesmo POST sem medo. Útil em retry de timeout/5xx.

Espelha o handler interno PATCH /api/procedimentos-tarefas/:id/resultado descrito em §5.3 do doc 01.

8.1 Endpoint
PATCH /api/integracoes/betshark/procedimentos/:external_id/resultado

8.2 Body
{
  "resultado_lucro": "0.00",
  "resultado_freebet_ganha": "50.00",
  "freebet_creditada": "SIM",
  "resultado_observacao": "Real venceu 2x1, FB de R$50 caiu na conta."

Campo	Tipo	Obrigatório?	Notas
resultado_lucro	decimal string	Sim	Cash real. Aceita positivo, zero ou negativo.
resultado_freebet_ganha	decimal string	Não (default "0")	Valor da FB ganha confirmada.
freebet_creditada	"SIM" | "NAO" | null	Quando resultado_freebet_ganha > 0	Se null/ausente com FB > 0, força "NAO".
resultado_observacao	text	Não	Observação livre.
8.3 Comportamento server-side (igual ao handler interno)
Backend:

Grava resultado_lucro, resultado_freebet_ganha, resultado_observacao.
Por compat com código legado, escreve cenario_realizado = 'A' | 'B' (A se FB=0, B se FB>0).
Decide status_operacao:
LUCRO_DIRETO se não houve FB ganha ou se freebet_creditada='NAO'.
FALTA_GIRAR_FB quando resultado_freebet_ganha > 0 E freebet_creditada='SIM'.
Grava freebet_creditada exatamente como veio (quando FB=0, força 'NAO').
Emite SSE pros clientes Pro recarregarem o catálogo.
8.4 Resposta
{
  "ok": true,
  "data": {
    "internal_id": 12345,
    "status_operacao": "FALTA_GIRAR_FB",
    "synced_at": "2026-05-03T22:15:00.000Z"
  }
}

⚠️ Igual ao handler interno: este endpoint não cria linhas em freebets (inventário pessoal por cliente). Esse comportamento está documentado em §7 e §9 do doc 01.

11. Envelope de erro padrão
Todo erro segue o mesmo formato:

{
  "ok": false,
  "error": {
    "code": "VALIDATION_ERROR",
    "message": "Campo 'categoria' inválido.",
    "field": "categoria",
    "request_id": "req_01HZX..."
  }
}

HTTP	code	Quando acontece
400	VALIDATION_ERROR	Body malformado, campo desconhecido fora da whitelist, URL inválida, decimal inválido.
401	UNAUTHORIZED	Auth (chave / assinatura / timestamp) falhou. Sem detalhe.
403	FORBIDDEN_PARTNER	Parceiro ativo=false, ou IP fora da allowlist.
404	NOT_FOUND	external_id não existe (em PATCH/GET/arquivar).
409	CONFLICT	Race condition em upsert (raro; tentar de novo).
422	UNPROCESSABLE_PAYLOAD	Mapeamento de enum falhou (categoria/tipo com valor não suportado).
422	UNRESOLVED_REFERENCE	external_referencia_id aponta pra UUID inexistente no nosso lado.
429	RATE_LIMITED	Mais de rate_limit_rpm chamadas no minuto. Header Retry-After indica segundos.
500	INTERNAL_ERROR	Erro nosso. Vocês podem retry com backoff.
503	MAINTENANCE	Janela de manutenção rara. Header Retry-After indica segundos.
Todo response (sucesso ou erro) traz o header X-Request-Id pra correlação com nossos logs. Quando vocês reportarem um problema, mandar esse request_id acelera o suporte.

12. Operacional — rate limit, segurança, logs
12.1 Rate limit
Default: 60 requisições por minuto por API key. Configurável por parceiro.
Janela deslizante (sliding window).
Excedeu → 429 com Retry-After: <segundos>.
12.2 IP allowlist
Opcional. Vazio = qualquer IP.
Quando preenchida (lista de IPs ou CIDRs), chamadas de fora dela retornam 403 FORBIDDEN_PARTNER.
12.3 Tamanho do body
Limite de 32 KB. Body maior → 400 VALIDATION_ERROR. Não estamos esperando payload grande aqui.
12.4 Validação de schema
Zod estrito (.strict()). Qualquer campo extra fora da whitelist é rejeitado em vez de silenciosamente ignorado.
12.5 Logs e auditoria
Toda chamada vai numa tabela integracoes_log com: parceiro_id, rota, method, external_id, http_status, latency_ms, ip, request_id, created_at. Sem corpo, sem segredos.
Retenção mínima: 90 dias.
Útil pra responder reclamações tipo "criei e não apareceu".
12.6 Política de retry sugerida (lado de vocês)
Em 5xx ou erro de rede: retry exponencial com jitter (1s, 2s, 4s, 8s, 16s, max 5 tentativas).
Em 409 CONFLICT: retry imediato 1x.
Em 429: respeitar Retry-After.
Em 4xx (exceto 409/429): não retry — é erro de contrato, abrir bug.
13. Pontos resolvidos (rodada 2026-05-03)
Todas as 8 perguntas abertas da rodada inicial foram fechadas com a BetShark.

Slug do parceiro: ✅ 'betshark' confirmado.
Homolog separado: ✅ vamos expor app-staging.freebetpro.com.br. A BetShark roda 5–10 procedimentos reais (criar / atualizar / arquivar / definir resultado) contra esse host antes da virada pra prod.
Owner único vs multi-owner: ✅ 1 parceiro = 1 owner_client_id no MVP.
external_referencia_id (cadeia FB cross-system): ✅ confirmado. A BetShark vai criar do lado deles procedures.freebet_reference_id uuid REFERENCES procedures(id) e popular gradualmente; o campo texto livre freebet_reference continua existindo lá pra retrocompat. No payload pra cá: mandam external_referencia_id quando o link UUID existe; null quando o gerente só digitou texto livre. Caso de queimar FB cuja origem é outro sistema (não BetShark) fica fora do MVP.
numero na resposta: ✅ vamos sempre devolver. A BetShark vai provisionar procedures.freebetpro_numero int (nullable, aditiva) e exibir como "Catálogo FreeBet Pro #N" no card e em relatórios deles.
Webhook reverso (nosso → BetShark): ✅ fora do MVP, segue como follow-up. Quando entrar, eles expõem freebetpro-webhook em Supabase Edge Function com mesmo padrão Bearer + HMAC (simétrico).
Canal pra entregar a chave: ✅ Bitwarden Send (link expirável, leitura única). 1Password aceito como fallback. Email/Slack/Telegram não.
Janela de timestamp HMAC = 5 min: ✅ folgada. Lado deles é Supabase Edge Functions (Deno + NTP do provider) com clock skew típico < 1s.
13.1 Ajustes resultantes do retorno da BetShark
a) Mapeamento de titulo (lado deles, não muda nosso contrato): a BetShark não tem campo dedicado pra título. O cliente HTTP deles vai concatenar ${platform} — ${promotion_name} antes de enviar. Pra nós, continua chegando como string única no campo titulo. ✅

b) Mapeamento de categoria (lado deles): os valores em produção deles batem com nosso enum (Promoção → PROMOCAO, Cashback → CASHBACK, Freebet → FREEBET, Superodd → SUPERODD, Extra → EXTRA, Ganhar Giros Gratis → GIROS_GRATIS). DUPLO_GREEN não é categoria pra eles — vai como tag via dp=true. ✅

c) prioridade: a BetShark não tem o campo. Vão sempre mandar 'MEDIA' (default). Se um dia precisarem, viram do lado deles. ✅

d) descricao_promocao e link_promocao: ambos opcionais e nullables, sem warning quando ausentes. Já refletido na tabela §6.3 acima.

e) IP allowlist: ✅ vazia pra betshark no MVP. O outbound deles vem de Supabase Edge Functions (us-east-1, IPs dinâmicos do Cloudflare/AWS — sem IP fixo). Auth fica 100% no Bearer + HMAC, que já é forte. Se um dia quisermos trancar por IP, eles migram pra worker com IP fixo — não vale fazer agora.

f) Limite operacional confirmado: picos esperados deles ~20 chamadas/min em horário de pico (Sábado 14h–18h, lote pré-jogo). 60 rpm de default tem 3x de headroom. ✅

g) X-Request-Id: a BetShark vai guardar do lado deles em procedures.freebetpro_last_request_id pra correlação rápida quando reportarem problemas. Nosso lado não muda — já mandamos esse header em toda resposta. ✅

13.2 POST upsert vs PATCH parcial — divisão de responsabilidades
A BetShark perguntou explicitamente como dividir os endpoints. Confirmação oficial:

POST /procedimentos = criar e editar qualquer campo da §6.3 (idempotente por external_id). Se a única coisa que mudou foi o título ou a casa_aposta, manda POST mesmo. Vai fazer upsert.
PATCH /procedimentos/:external_id = atalho opcional pra atualização parcial quando o cliente HTTP já sabe que o registro existe. Não é obrigatório usar. A BetShark optou por sempre usar POST upsert pra evitar a corrida de "criar + PATCH antes do create propagar". Funciona.
PATCH /procedimentos/:external_id/resultado = exclusivo pro modal "Definir Resultados". Não dá pra setar resultado_lucro/resultado_freebet_ganha/freebet_creditada via POST nem via PATCH normal. Tem que vir por aqui.
POST /procedimentos/:external_id/arquivar = exclusivo pra arquivar/restaurar.
Resumindo o fluxo da BetShark: POST upsert em todo save do ProcedureModal; PATCH /resultado quando salvarem o DefinirResultadosModal; POST /arquivar quando clicarem em arquivar/restaurar.
