# BetShark Pro

## Overview
BetShark Pro is a real-time odds monitoring application for sports betting. It features dashboard views, detailed match information, procedure control, subscriptions, betting affiliate tools (BetBra), and administrative panels. The project aims to provide a comprehensive, real-time sports betting odds platform with a focus on user engagement and conversion through a sophisticated trial system.

## User Preferences
- Language: Portuguese (Brazilian) - all UI text is in Portuguese
- Dark theme preferred (default)
- UI style: "Instagrammable" — neon accents, glassmorphism, gradient text, animated elements

## System Architecture
- **Frontend**: React 18, TypeScript, Vite, styled with Tailwind CSS and shadcn/ui.
- **Backend**: Supabase (auth, database, realtime) and Supabase Edge Functions for business logic and integrations.
- **State Management**: @tanstack/react-query for server state, React Context for authentication.
- **Routing**: react-router-dom v6 with framer-motion for animated transitions.
- **UI/UX**:
    - **Theme**: Dark mode by default, featuring a deep navy background, neon green primary color (`hsl(145 80% 48%)`), and radial green glow effects.
    - **Typography**: Inter and Space Grotesk for sans-serif, JetBrains Mono for monospace.
    - **Components**: Utilizes custom utility classes like `gradient-text`, `glow-primary`, `glass`, `animate-fade-in-up`, and `card-hover` for a dynamic and engaging aesthetic.
    - **Layout**: Consistent page headers with icon badges, titles, and subtitles. A sidebar includes a gradient logo mark with a pulse dot and color-coded navigation icons.
- **Trial Telegram System**:
    - Manages lead acquisition and trial activation through Telegram.
    - Implements a forced bot interaction flow to ensure direct messaging capability.
    - Sends automated 24h and 1h expiration reminder DMs with customizable coupons.
    - Supports an optional second "bonus" Telegram group for enhanced user engagement.
    - Features a robust anti-repeater mechanism based on `telegram_user_id` to prevent duplicate trials.
    - **Paid Re-entry (Lastlink)**: When an expired/removed lead is re-added to the VIP group via an external invite_link (e.g., generated by Lastlink) or by another bot/admin (`chat_member.from.id ≠ user.id`), `trial-webhook` skips the re-kick and marks the lead as `status='converted'`. This prevents our bot from kicking customers who paid and were re-added by Lastlink's bot. Allowed status values: `pending|active|expired|removed|blocked|blocked_repeat|converted` (migration `20260501_trial_status_converted.sql`).
    - Provides an admin panel (`/trial-admin`) for lead management, diagnostics, and manual interventions.
    - Incorporates structured logging for all critical webhook decisions.
    - **Invite link GC (cota do Telegram)**: o Telegram limita o número de invite_links ATIVOS por chat por bot. Como o `trial-signup` cria 1 link novo por cadastro (member_limit=1, expire_date=now+24h) e links que expiram naturalmente continuam contando pra cota até serem REVOGADOS, esse teto era atingido em produção e a LP parava de gerar links (sintoma: "Não foi possível gerar o link do grupo no momento"). Solução: `supabase/functions/_shared/link-gc.ts` (`runLinkGc`) revoga via `revokeChatInviteLink` todo `invite_link`/`bonus_invite_link` de leads com `created_at < NOW()-24h`, independente do status, e zera as colunas. Idempotente (links 404/expirados pelo Telegram contam como ok). Rodado em duas pontas: (1) `trial-cron` chama no fim de cada tick (a cada 30min), (2) `trial-link-gc` Edge Function exposta com botão "Limpar links antigos" no dialog de diagnóstico do `/trial-admin` (auth: JWT do admin OU service-role). Migração: nenhuma — só mexe em colunas existentes.
    - **Scheduling**: `pg_cron` + `pg_net` extensions on Supabase invoke the `trial-cron` Edge Function every 30 minutes (job `trial-cron-every-30min`). The function accepts two bearer tokens (defense-in-depth): the auto-injected `SUPABASE_SERVICE_ROLE_KEY` and a custom `TRIAL_CRON_SECRET` Edge Function secret. The custom secret is stored in Supabase Vault as `trial_cron_secret` and read by the cron at runtime via `vault.decrypted_secrets` — this decoupling keeps the cron working even if the platform silently rotates the auto-injected service role key. Migration: `supabase/migrations/20260501_trial_cron_schedule.sql`. The cron only processes leads with `cohort='v2'`; legacy `v1` leads are intentionally excluded.
- **Landing Page (LP Shark 100% Green)**: A rebranded landing page (`/trial`) with a luxury dark-green aesthetic, multiple CTAs for trial signup, free groups, and direct purchase, all tracking events for conversion analytics.
- **Performance tracking pixels**: Two pixels load on every page (configured in `index.html` `<head>`):
    - **Meta Pixel** (`fbq`, ID `1486249896237184`) — initialized on the LP via `TrialLanding.tsx` with CAPI deduplication.
    - **track4you** (`https://track4you.app/9fdc319a-0e6e-4feb-b4a4-471c0293465c.js`, deferred) — for Arthur (gestor de performance). Click events are dispatched on the 3 Telegram-bound CTAs of the LP via the `trackT4Y()` helper in `TrialLanding.tsx` (event name `cta_telegram`). The helper is defensive: it tries multiple known global names (`window.t4y`, `window.track4you`, `window.T4Y`) and also dispatches a `t4y:event` `CustomEvent`, falling back to no-op if the SDK hasn't loaded yet — so it never blocks the redirect.
- **Data Cohorting**: Implements a 'v1'/'v2' cohort system to manage distinct user groups, particularly for handling legacy data without affecting active trial flows.
- **Procedimentos — Paridade FreeBet Pro (§8.1–§8.4)**: O painel `/procedure-control` agora bate com o FreeBet Pro nas funcionalidades essenciais (a §8.5, sync por endpoint público, está pendente até a spec do endpoint). Migration aditiva: `supabase/migrations/20260503_procedures_paridade_freebetpro.sql`. Decisões fechadas: (1) `dp` boolean fica e mapeia pra tag "Chance Duplo" na hora do sync; (2) previsional × realizado separados — `lucro_prejuizo_previsto`/`freebet_valor_previsto` no momento da criação, `resultado_lucro`/`resultado_freebet_ganha`/`freebet_creditada` (SIM|NAO) no momento da conferência; backfill `resultado_lucro := profit_loss` pra não perder dados antigos. As colunas legadas `profit_loss`/`freebet_value` continuam sendo escritas em espelho pra preservar gráficos/KPIs existentes. (3) Ação **Arquivar** (soft delete via `archived` bool + `archived_at`) coexiste com Excluir (hard delete) — toggle "Mostrar arquivados" nos filtros, index parcial `idx_procedures_not_archived` pro caminho quente. (4) `partida_descricao` é texto livre no MVP (sem `fixture_id`). (5) UUID `id` será o `external_id` quando o sync da §8.5 entrar (`freebetpro_external_id`/`freebetpro_synced_at`/`freebetpro_last_error` já provisionados). UI: novo modal **DefinirResultadosModal** abre pelo botão "Conferir" (só aparece após kickoff+150min, helper `src/lib/procedureGameTime.ts`), auto-decide o status — `Lucro Direto` se sem FB previsto OU `freebet_creditada=NAO`; `Falta Girar Freebet` se FB>0 E `freebet_creditada=SIM`. ProcedureModal de criação ganhou seção "Jogo/Evento" (data, horário, descrição), radio de "Tipo" (Sem FB / Ganhar FB / Queimar FB) substituindo a seção Freebet condicional, e sugere o próximo Nº livre. Filtros novos: toggle "Mostrar arquivados" + select "Horário do jogo" (Em breve / Ao vivo ≤150min / Encerrado / Sem partida).

## Production Hosting
- **Deployment**: `autoscale` (mudado de `static` em mai/2026 pra suportar SPA fallback). Build: `npm run build` → `dist/`. Run: `node server.cjs`.
- **`server.cjs`**: Servidor estático minimalista em Node puro (zero deps). Serve `dist/` e faz fallback pra `dist/index.html` em qualquer rota desconhecida — sem isso, `Ctrl+F5` em deep links como `/lastlink-admin` retornava 404 no app publicado. HTML é servido com `Cache-Control: no-store` (deploy reflete na hora) e assets hashados com `immutable` (cache forte). Bloqueia path traversal.

## Realtime & UI Stability
- **Debounce no `useRealtimeSubscription`** (`src/hooks/useRealtimeSubscription.ts`): coalesce de 400ms (default) nas invalidações. Sem isso, um burst de eventos (cron, webhook em lote, admin editando várias linhas) disparava N refetches simultâneos e cada cliente conectado piscava várias vezes seguidas.
- **`placeholderData: (prev) => prev` por hook (NÃO global)**: aplicado apenas em `useTrialLeads`, `useLastlinkPayments`, `useLastlinkEvents`, `useLastlinkLeadEvents` — as listas grandes que rebuscam por interval/realtime. Mantém os dados antigos visíveis durante refetch pra UI não piscar pro skeleton e o usuário não perder scroll/drawer/foco. **Importante**: tentamos fazer isso como default global no QueryClient (`src/App.tsx`) e quebrou filtros client-side de forma sutil (input de busca do TrialAdmin parou de funcionar). A regra é: aplicar local nas queries pesadas, nunca global.

## External Dependencies
- **Supabase**: Primary database, authentication, and realtime services. Utilizes multiple Supabase instances for different data domains (main data, procedures).
- **Telegram Bot API**: For managing trial leads, sending direct messages, handling group invites, and processing `chat_member` webhooks.
- **Vite**: Frontend build tool.
- **Tailwind CSS**: Utility-first CSS framework for styling.
- **shadcn/ui**: UI component library.
- **react-router-dom**: Declarative routing for React.
- **framer-motion**: Animation library for React.
- **@tanstack/react-query**: Data fetching and caching library.
- **date-fns**: JavaScript date utility library.
- **Lastlink Admin Page (`/lastlink-admin`)**: Painel completo que desmistifica todos os dados capturados pelo webhook. Lista pagantes (filtra por status, método, cupom, afiliado, plano, UTM source, período custom, esconde testes), 6 KPIs (receita, conversões, ticket médio, refunds, top cupons, top afiliados), drawer detalhado por cliente com 7 seções (header, contato+endereço, pagamento atual, IDs Lastlink, marketing/UTM, status&datas, timeline de eventos, raw payload colapsável) e tab "Eventos brutos" com filtros de tipo/matched. Realtime ativo em `trial_leads` e `lastlink_events`. Acessível via sidebar (item "Pagamentos Lastlink", ícone `Receipt`, cor emerald) e via link discreto "Detalhes do pagamento" no `TrialAdmin` para leads com `status='converted'` ou `paid_at` (passa `?lead=<id>` para auto-abrir o drawer).
- **Lastlink**: External checkout platform integrated for subscription payments and coupon application. The `lastlink-webhook` Edge Function (`supabase/functions/lastlink-webhook/index.ts`, v2) receives Postback events from Lastlink at `https://wspsuempnswljkphatur.supabase.co/functions/v1/lastlink-webhook?token=<LASTLINK_WEBHOOK_TOKEN>` (token stored as Edge Function secret). It saves every raw event to the `lastlink_events` audit table (with `is_test` flag for events triggered by the "Testar" button or with `IsTest=true`) and matches the buyer to a `trial_leads` row by email/order_id/subscription_id. On match, it populates **24 fields**: payment data (`lastlink_order_id`, `lastlink_subscription_id`, `lastlink_payment_id`, `lastlink_offer_id/name/url`, `paid_amount`, `paid_currency`, `original_price`, `paid_at`, `payment_method`, `installments`, `plan_name`, `coupon_code`, `lastlink_invoice_url`, `lastlink_origin_url`, `lastlink_affiliate_email`, `recurrency_months`, `next_billing_at`), buyer data (`buyer_name`, `buyer_document` (CPF), `buyer_phone`, `buyer_address` jsonb), tracking (`lastlink_utm` jsonb), and status (`subscription_status`, `canceled_at`, `refunded_at`, `lastlink_last_event`, `lastlink_event_id`, `lastlink_is_test`, `lastlink_raw`). On paid events (`Purchase_Order_Confirmed`, `Purchase_Request_Confirmed`, `Recurrent_Payment`) it forces `status='converted'` to prevent the trial-webhook re-kick path. On cancel/refund events (`Subscription_Canceled`, `Subscription_Expired`, `Product_Access_Ended`, `Payment_Refund`, `Payment_Chargeback`, `Refund_Requested`) it updates `subscription_status` without kicking. Migrations: `supabase/migrations/20260502_trial_lastlink_payment.sql` (v1, base columns + audit table) and `supabase/migrations/20260502_trial_lastlink_extra_fields.sql` (v2, extra fields after inspecting real payload).
- **track4you**: 3rd-party performance/conversion tracking pixel (script loaded in `index.html`).
